# src/content 文章特点与写作方法（归纳自现有文章）

本文件基于当前仓库 `src/content/blog` 下的两篇文章内容进行归纳，目标是把「文章的共性特点」和「可复用的写作方法」落成一份可直接照着写的新文章指南。

参考来源（可直接打开对照）：

- `src/content.config.ts`
- `src/content/blog/computer-organization-101/index.mdx`
- `src/content/blog/data-representation-102/index.mdx`
- `src/pages/index.astro`
- `src/pages/blog/[id].astro`
- `astro.config.mjs`
- `src/styles/global.css`

## 1. 内容系统与约束（写之前必须知道）

### 1.1 内容组织方式（目录结构）

当前内容采用「每篇文章一个文件夹」的组织方式：

- `src/content/blog/<slug>/index.mdx`
- 资源（图片等）通常放在 `src/content/blog/<slug>/picture/`

其中 `<slug>` 会直接参与：

- 列表页卡片的 `data-post-id`（见 `src/pages/index.astro`）
- 文章页路由 `/blog/<slug>`（见 `src/pages/blog/[id].astro`，它会把 `entry.id` 的 `/index` 去掉作为 `id`）

建议约束：

- `<slug>` 只用一层目录，不要出现 `a/b` 这种带斜杠的 id（否则无法匹配 `/blog/[id]` 这种单段路由）。
- `<slug>` 建议使用 `kebab-case`（小写 + 连字符）。

### 1.2 Frontmatter（元信息）Schema

`src/content.config.ts` 对 `blog` collection 约束如下：

- 必填：`title`（string）、`date`（date）
- 可选：`description`（string）、`tags`（string[]）、`image`（string）

其中 `date` 当前写法是 YAML 日期（例如 `2026-04-10`），会被解析为 `Date`，符合 `z.date()`。

当前页面实际用到的字段（见 `src/pages/index.astro` 与 `src/pages/blog/[id].astro`）：

- `title`：必须有
- `date`：必须有
- `description`：可选，存在则在标题下展示
- `tags`：可选，存在则展示为标签胶囊
- `image`：目前未在页面里使用（可预留给未来的列表封面/OG 图/卡片背景等）

### 1.3 MDX 能力边界（项目已启用的特性）

`astro.config.mjs` 已启用：

- `@astrojs/mdx`
- `remark-math` + `rehype-katex`（数学公式）

`src/styles/global.css` 已导入：

- `katex/dist/katex.min.css`（KaTeX 样式）
- `@tailwindcss/typography`（文章区域使用 `prose` 排版）

结论：当前文章可以稳定使用：

- Markdown 基础语法（标题/列表/表格/代码块/引用/分割线）
- LaTeX 数学公式：行内 `$...$`、块级 `$$...$$`
- MDX：`import` 静态资源、直接写 JSX/HTML 结构（如 `<figure>`、`<img>`）

### 1.4 页面呈现方式对写作的影响（这不是纯 Markdown 站点）

这个博客的阅读入口有两个：

- 列表页（`/`）：卡片网格 + 点击展开的“沉浸阅读”面板（见 `src/pages/index.astro` + `src/scripts/post-expander.js`）
- 文章页（`/blog/<id>`）：传统文章页（见 `src/pages/blog/[id].astro`）

对写作的直接影响：

- `description` 的地位很高：它会在列表页展开面板顶部展示，读者往往先读它再决定是否继续，所以它应该像“导语/摘要”，而不是“关键词堆叠”。
- 内容默认会包在 `prose prose-lg ...` 的排版体系里（Tailwind Typography）。这意味着：
  - 标题/段落/表格/代码块都有统一的排版节奏，尽量用“标准 Markdown 结构”来让样式自然生效。
  - 如果要做“全宽/定制布局”的内容块（比如图、并排对照、特殊提示框），用 MDX/HTML，并配合 `not-prose`（参考现有 `<figure class="not-prose">` 写法）。
- 列表页是“先扫读再深入”的阅读路径：读者会先扫 `title/date/description/tags`，再进入正文，所以这些元信息要能独立成立。

## 2. 现有文章的共性特点（写出来的“样子”）

### 2.1 叙事主线：第一性原理 + 问题驱动

两篇文章的核心写法不是“背概念”，而是：

1. 先从一个朴素的需求/困惑出发（读者真实会问的问题）
2. 按因果链条逐个遇到障碍（为什么不行？哪里卡住？）
3. 给出设计/编码/约定作为回答（为什么这样设计？有什么后果？）
4. 用例子验证（算一遍、画一遍、跑一遍、对照一遍）
5. 用“小结”把线索收束成一张表/几条结论

这种结构的好处：

- 读者能记住“为什么”，而不是只记“是什么”
- 章节之间可以自然衔接（上一章提了什么，本章就补什么）

### 2.2 结构层面：编号式章节 + 频繁的局部总结

常见结构形态：

- 大标题多为系列化命名：`计算机组成原理 101/102：…`
- 正文以显式编号组织：`## 1.1`、`### 1.2.1` 等
- 使用 `---` 横线做“章节段落的节拍器”（切场/收束/过渡）
- 频繁出现 `### 小结` / `## 小结`：用表格把关键信息压缩成可回忆的结构

读者体验特点：

- 每一个小节都能单独读懂，并且读完有“收口”
- 章节编号让长文不容易迷路，回看也更可定位

### 2.3 语言层面：对话式提问 + 关键句加粗

典型语言手法：

- 用直接提问把读者的注意力“锁”在当前要解决的具体困惑上（问题往往短、直、像脑内自问自答）
- 对关键概念用加粗做“视觉锚点”：`**补码（Two's Complement）**`
- 中英术语并列（尤其是首次出现）：中文名 + 缩写 + 英文全称
- 对抽象概念给直觉比喻：公寓楼/里程表/刻度尺/复习 408 的三条指令

写作上要点：

- 加粗不是装饰，而是“让读者扫读也能抓住骨架”
- 术语并列要克制：只在首次出现或必要对照处写全，后续用缩写即可

### 2.4 表达层面：表格、代码块、公式、步骤推演

两篇文章里高频出现的“讲清楚工具箱”：

- 表格：对比、归纳、列举（范围/特点/差异）
- 代码块：展示流程（取指译码执行）、展示示例代码（C 的编译流程）
- 数学公式：定义关系（主频与周期、CPI、MIPS 等），并配文本解释
- 步骤推演：用列表把一个转换/算法走完（十进制转二进制、乘 2 取整等）

小技巧：

- 讲公式时尽量“先直觉，再公式，再带入例子验证”
- 步骤推演尽量让读者能跟着算出同样结果（并用 `✓` 标记校验点）

### 2.5 语气与叙事：严谨但不学院派（像在带着读者一起推演）

这两篇文章的“声音”有一些稳定特征：

- 第一人称复数主导：用“我们”把读者拉进同一个推理过程（不是“我讲你听”，而是“一起把问题拆开”）。
- 口语化连接词密度高（推进、转折、收束的节拍很明显）。它们不是废话，而是在给读者做“思维换挡提示”。
- 结论句敢落地：不回避判断（例如“这显然不可持续”“这就是为什么……”），但会紧跟解释与因果链条来支撑。
- 对读者的预期很明确：默认读者会写点代码、知道十进制/二进制这些基础；但不假设读者已经理解硬件内部细节，所以会从最小动机推到结构。

写作时可复用的语气策略：

- 先把读者的疑问说出来，再回答它。
- 每给出一个术语或设计，立刻解释“它解决的具体麻烦是什么”。
- 尽量用“如果……那就……”这种条件链来表达因果，而不是堆概念定义。

### 2.6 段落节奏：短段落 + 强对比 + 明确切场

段落节奏非常关键，因为文章会在“卡片展开阅读”里被连续滚动阅读：

- 短段落（1-3 句）居多，降低读者的工作记忆负担。
- “对比句”密度高：常见的是转折、利弊拆分、直觉与事实对照（让读者不断校正自己的理解）。
- 用 `---` 频繁切场：通常出现在“完成一个阶段”或“准备换一个子问题”的地方，起到“停顿一下、换档继续”的作用。
- 局部标题非常细：让读者可以跳读，并且在回看时快速定位。

可执行建议：

- 每讲完一个小推演（尤其是带表格/代码块的部分），紧跟一段 1-2 句的“解释它意味着什么”。
- 当同一小节里出现第 3 次“但/所以”，通常意味着该拆成新的 `###` 了。

### 2.7 视觉标记（让扫读也能抓住骨架）

现有文章的视觉标记手段比较克制但有效：

- 加粗用于“骨架词”：核心术语、关键判断、必须记住的结论。
- 引用块（`>`）用于“核心思想/一句话定性”：例如存储程序思想那句。
- 行内代码（`` `...` ``）用于“代码/字面量/关键词”：寄存器名、指令名、十六进制片段等。
- 表格用于“压缩记忆”：把一大段解释缩成对照关系。

一个容易被误判为“Markdown 写错了”的细节：

- 这个站点用了 `@tailwindcss/typography` 的 `prose` 排版，它默认会在行内 `code` 的前后用 CSS 伪元素塞一对反引号（看起来像 `` `a` `` 这种）。
- 所以你在正文里写了行内代码，页面上多出一对反引号，不是渲染器抽风，是样式在“装饰”。

解决方式（推荐在站点层面一次性修好，而不是在文章里绕）：

- 在 `src/styles/global.css` 覆写掉 `code::before/after` 的 `content`（仓库里已经加了这段覆盖）。
- 如果你想保留行内等宽字体，但不想看到“反引号壳子”，这就是最干净的做法。

写作侧的小经验（即使样式修了也有用）：

- 短 token（寄存器名、类型名、常量）适合行内代码。
- 长位串/二进制片段更适合用代码块，让对齐和换行可控，读者也更容易扫。

反模式（建议避免）：

- 一段里加粗过多，读者会失去“重点感”。
- 用表格塞长段文字，移动端难读。
- 用长列表堆定义而不举例，读者会看完但无法复述。

### 2.8 “例子即验证”：用可复算的样例建立信任感

两篇文章都在用一种“几乎像单元测试”的策略：

- 选小而典型的数字（42/200/255/0.625）让读者可以手算。
- 每推导一次就验证一次：要么算回去，要么对照范围，要么展示溢出丢弃如何得到正确结果。
- 用 `✓` 标记验证点，让读者知道“这里算对了，可以继续往下读”。

可执行建议：

- 每个核心概念至少配 1 个“能算出来的例子”。
- 例子不要追求复杂，追求“读者能跟着走完且不出戏”。

### 2.9 视角移动：不断在“大图景”和“细部机制”之间来回切换

这两篇文章读起来不像一条直线，更像镜头在切换焦段：

- 先在高处把要解决的麻烦画出来（为什么会卡、卡在哪里、这件事会影响什么）
- 然后迅速下潜到一个可以操作的层级（寄存器、编码规则、字节序、公式）
- 讲完细节立刻抬头：把细节放回系统里，解释它改变了什么、带来了什么副作用、下一步会遇到什么新麻烦

这种写法的微妙之处在于：细节不是“为了完整”，而是“为了让上层判断更有抓手”。读者不会觉得你在背书，而会觉得你在把抽象变成可以拿来做推理的零件。

### 2.10 “先不讲”的克制：把复杂性推迟，但不让读者觉得被糊弄

文章会经常遇到这种情况：为了把当前问题讲清楚，你不得不提到一个更复杂的概念（例如缓存、浮点数的完整格式、微架构的机制）。现有文章的处理方式很成熟：

- 不回避它的存在，但只给“它解决的矛盾是什么”这一点
- 明确告诉读者：现在先把它当作一个黑盒/先知道它在这里就够了
- 给一个轻量的承诺：后续章节会把它拆开讲

这种克制的核心不是省略，而是“控制读者的认知负债”。你让读者知道你没有忘记，只是现在不该把注意力摊薄。

### 2.11 术语的生命周期：首次锚定、后续放手、必要时再召回

术语处理的风格很像写一份可长期维护的笔记：

- 首次出现时把概念锚住：中文名、缩写、英文名（按你希望读者未来检索的方式）
- 之后不反复念全称，用缩写或中文名就够了，让阅读更轻
- 当章节跨度很大或概念容易混淆时，再用一句话把它召回（而不是把定义整段贴回来）

这种处理会让文章同时具备两种品质：首次阅读顺滑、二次查阅好定位。

### 2.12 可信度来自可审计：范围、边界、反例、以及“同一比特序列不同解释”

这两篇文章很少“空口讲结论”，它们更像在做可审计的推导：

- 给出明确范围与单位（位宽、字节、指数范围、CPI/主频的量纲）
- 主动展示边界与坑点（比如某些十进制小数在二进制里不可精确表示、有符号补码多出一个负数）
- 把“解释”与“演算/对照”绑定：读者能跟着算一遍，或者至少能对照表格复核
- 强调解释与表示的分离：同一段比特在不同语境下的含义不同，这种提醒会显著减少误解

这种写法带来的感觉是：你不是在输出知识，而是在展示一套推理方法，读者可以拿去复用。

### 2.13 文字不是装饰：每段都有明确的任务

它们的段落通常不会承担“气氛段”的职责，更多是高密度的功能段：

- 推进：把问题往前推一格
- 钉死：把一个定义或结论钉牢
- 过桥：把当前结论连接到下一个麻烦
- 缓冲：让读者喘一口气，同时确认理解没有跑偏

如果一段话既不推进也不钉死，读者会在卡片展开阅读里快速滑走。现有文章整体上避免了这种“无任务段落”。

### 2.14 微观排版：空行、短句、以及“把问题单独放一行”

即使不看内容，只看排版节奏，也能感到它在控制阅读速度：

- 关键问题经常被单独放在一行或一个短段里，让读者在滚动中自然停顿
- 复杂推导前会留出空行、插入小标题或分割线，给读者一个重新对齐的机会
- 表格/代码块前后通常有解释段，用来告诉读者“你现在应该看什么、看完应该得到什么”

这种做法会让文章更像“带节拍的讲解”，而不是“没有呼吸的密集笔记”。

## 3. 现有文章的 MDX/排版写法（怎么写出来）

### 3.1 Frontmatter 写法（字段意图与写作侧重点）

建议：

- `title`：长标题没问题，但最好能在列表页一眼读出“研究对象”和“推进方向”，像章节标题而不是宣传语
- `description`：它在列表页/文章页都会以“导语”的位置出现，应该像摘要一样可独立成立，读完就知道这篇要解决什么、怎么解决、以及读完会更清楚什么
- `date`：保持 `YYYY-MM-DD`（不要写成字符串，避免解析偏差）
- `tags`：更像“课程目录/索引卡”，不要把它当成 SEO 关键词堆栈；少而准比多而全更贴合这个博客的气质

### 3.2 标题层级与编号（建议约定）

现有文章默认遵循：

- `##`：章/大节
- `###`：小节
- `####`：局部重点/直觉/补充

建议延续的编号方案：

- 大节：`## 3.1 xxx`
- 小节：`### 3.1.1 xxx`

编号的价值在于可引用、可回看、可分块。

### 3.3 数学公式（KaTeX）

行内：

```md
主频与时钟周期互为倒数：$\text{主频} = \frac{1}{\text{时钟周期}}$
```

块级：

```md
$$
\text{CPU 执行时间} = \frac{\text{指令条数} \times \text{CPI}}{\text{主频}}
$$
```

建议：

- 公式尽量搭配一句“这是什么意思/哪几个变量可优化”
- 块级公式上下留空行，避免与段落粘连影响可读性

### 3.4 代码块（流程与示例）

建议给代码块加语言标识（便于高亮）：

```c
// hello.c
#include <stdio.h>
int main() {
  printf("hello, world\n");
  return 0;
}
```

流程类（不一定需要语言标识，关键是对齐与可读性）：

```text
取指（Fetch）→ 译码（Decode）→ 执行（Execute）→ ...
```

### 3.5 表格（对比/归纳/范围）

现有文章的表格特点：

- 常用居中对齐（`:---:`）强化“对照感”
- 适合做“范围/特性/差异/步骤结果”的压缩表达

示例（强调对照关系与可回忆结构）：

```md
| 编码 | 8 位范围 | 零的个数 | 加减法 |
|:----:|:-------:|:--------:|:------:|
| 原码 | -127~+127 | 2 | 需要分情况 |
| 补码 | -128~+127 | 1 | 完全统一 |
```

### 3.6 图片（推荐：MDX import + 自定义 figure）

现有文章采用的稳定写法：

```mdx
import demoPng from './picture/demo.png';

<figure class="not-prose" style="margin: 1.75rem 0 2.15rem;">
  <img
    src={demoPng.src}
    alt="用一句话说明这张图在表达什么"
    loading="lazy"
    decoding="async"
    style="display:block;width:100%;max-width:720px;height:auto;margin:0 auto;"
  />
</figure>
```

要点：

- `not-prose`：让该块不被 `prose` 排版规则强行改样式（尤其是宽度/边距）
- `alt`：必须写，且要描述“图的语义”而不是“图的文件名”
- 建议统一 `max-width`，避免图片在大屏上过宽导致阅读负担

### 3.7 中英混排与标点（统一风格，减少阅读摩擦）

现有文章的混排倾向是“中文为主，英文只在必要处出现”：

- 术语首次出现：中文名 + 缩写 + 英文全称（或英文名）
- 后续：优先用缩写/中文名，保持阅读流畅
- 代码/寄存器/指令/字面量：用行内代码包裹

建议约定（尽量一致即可）：

- 英文缩写与中文之间留空格：`CPU`、`ISA`、`CPI`（写成 `CPU` 而不是 `ＣＰＵ` 或粘连在汉字里）
- 括号内容尽量短，负责“对照”而不是“再讲一遍”
- 引号风格统一：文章里既有直引号也有中文引号，建议后续统一用中文引号 `“”` 表示概念性引用，用英文引号 `""` 表示代码式字面引用

## 4. 写作方法：如何复刻这种文章（从 0 到 1）

这部分是“可执行的写法”，不是泛泛而谈。

### 4.1 选题与一句话目标（先锁边界）

写之前先把两件事想清楚（不需要写成固定说法，但需要能用一句话讲明白）：

- 这篇文章的主问题是什么，读者为什么会在这里卡住
- 读完之后，读者能多一种什么能力（能算、能判断、能解释因果、能把一段推导复述出来）

然后把主问题拆成 3~6 个“不可跳过的子问题”，每个子问题对应一个相对独立的小节，让读者每走完一段都能获得一个明确的中间结论。

### 4.2 每个小节的最小闭环（问题 -> 定义 -> 例子 -> 小结）

每个小节尽量完成一个闭环：

1. 抛问题：让读者知道你在回答什么
2. 给定义：用 1-3 句把概念钉死（必要时加中英术语）
3. 给例子：一步步算/对照表/画流程，保证可复现
4. 收束：用一张小表或 2-4 条 bullet 把结论压缩

如果小节写完读者还在问“所以呢？”，通常是缺了第 4 步。

### 4.3 “直觉段”与“验证段”的固定位置

现有文章很依赖一种“先把心智模型搭起来，再用算得出来的例子把它钉死”的节奏：

- 在关键概念附近插入一小段直觉解释（类比/模运算/里程表/刻度尺这类能直接形成画面的东西）
- 紧跟一个可复算的验证，让读者能在脑子里确认“是这么回事”，而不是只听你说“结论如此”

这在编码表示、进制转换、性能公式、抽象层次这类内容上尤其有效，因为它们天然容易变成“定义堆叠”。

### 4.4 小结的写法（把长文压成可回忆结构）

小结不写“总结一下我们讲了很多”，而是给读者一张“复习卡”：

- 用表格列出：问题 -> 方案
- 或者列出：概念 -> 作用 -> 关键点
- 或者列出：指标 -> 衡量什么

小结的目标是让读者 30 秒内能复述本文主线。

### 4.5 开头怎么写：三段式破题（让读者愿意继续）

现有文章的开头普遍遵循这种节奏：

1. 把“我们在解决什么”讲成一句朴素的人话
2. 立刻指出“看似简单但会连环撞墙”（制造张力）
3. 承诺本文路线：按顺序撞墙并逐个解决（给读者安全感）

### 4.6 过渡怎么写：把章节连成一条因果链

这个博客的一个显著风格是“章节不是并列百科条目，而是因果顺序”。

这种因果链的过渡不是靠固定说法，而是靠“问题的不可避免性”：

- 你在本节解决了一个麻烦，但解决方案本身会引出新的摩擦点（新的限制、速度瓶颈、表示不统一、实现复杂等）
- 你把这个摩擦点显式点出来，读者自然会跟着你走到下一节，因为他也想知道“那怎么办”

效果上，读者会感觉你不是在堆知识点，而是在带他沿着一条真实的工程路径往前走：每一步都合理、每一步都有代价、每一步都让下一步变得必要。

### 4.7 例子与反例：让读者“复现”你的推理，而不是“相信”你的结论

这个博客的说服力很大一部分来自例子的选择方式。它们往往满足几个很细的条件：

- 数足够小，读者愿意在脑子里算一遍或写两行验证
- 例子能暴露结构（比如位权、溢出、字节序），而不是只会得到一个正确结果
- 重要概念旁边往往会给一个“直觉会失败”的反例，让读者知道边界在哪里

写作上的效果是：读者不需要把你当权威，他可以自己把结论推出来。这会显著提升“读完之后能迁移”的程度。

### 4.8 张力管理：每一节都在解决麻烦，同时制造下一个麻烦

现有文章的推进感很强，因为它始终在做一种节奏控制：

- 你解决一个明确问题，读者得到一个确定的中间结论（有收获）
- 但你不会把故事停在“结论正确”，而是把结论的代价或副作用摆出来（新的麻烦）

这样读者会自然继续往下读，因为他不是在追一个“更大的知识树”，而是在追一个“还没闭合的因果链”。这也是长文能读完的关键。

### 4.9 简化与回补：先让模型能跑，再把真相补齐

技术文章很容易陷入两种极端：要么过早追求完整，把读者淹死；要么一直停留在比喻，导致读者无法落地。现有文章更像一种工程化的折中：

- 先给一个能跑的简化模型（用来解释当前问题）
- 明确指出简化的代价与盲点
- 在更合适的章节回补真实结构

这种写法会让读者在任何时刻都知道自己处于哪种抽象层级，从而减少“读着读着突然不知道自己在看什么”的失控感。

### 4.10 反模式清单（最容易把这种风格写崩的地方）

- 先堆一堆定义，再开始讲动机：读者会在第一个屏幕就流失。
- 章节顺序按“知识树”而不是“因果链”：看起来全面，但读起来不自然。
- 用“正确但没用”的结论句收尾：例如“总之很重要”。更好的做法是给出一个可复述的表格小结。
- 举例追求复杂：例子越复杂，读者越难验证，越不信服。

## 5. 发布前自检清单（避免上线后返工）

- 路径是否为 `src/content/blog/<slug>/index.mdx`，且 `<slug>` 不包含斜杠？
- Frontmatter 是否至少包含 `title`、`date`？`date` 是否是 `YYYY-MM-DD`？
- `description` 是否能在列表页当作摘要独立成立？
- 标题编号是否连续且层级清晰（不要乱跳 `##`/`####`）？
- 表格在移动端是否可读（列不要太多、单元格不要塞长段）？
- 图片是否有 `alt`？是否使用了 `not-prose`（需要时）？
- 数学公式是否渲染正确（`$` 与 `$$` 是否成对）？
- 关键结论是否在“小结”里可快速回忆？
- 开头 3-5 段是否能让读者清楚“本文要解决什么”？
- 每个核心概念是否至少配了 1 个可复算例子，并给出验证点？
- 加粗是否只用于“骨架词”（不滥用）？
